## 📌 Project Overview

This project simulates a **real-world trade monitoring workflow** for a financial markets platform. Using an AI-generated dataset modelled on actual trading data characteristics, the goal was to detect suspicious activity, quantify risk, and assign actionable alerts — replicating the kind of analysis done by fraud and risk teams at brokerages and trading platforms.

The analysis was structured around 5 independent challenges, each building on the previous to produce a complete trade risk scoring system.

---

## 🎯 Challenges & Solutions

### Challenge 1 — Flag Suspicious Trades
Identified anomalous trades by monitoring four key metrics: `lot_size`, `execution_latency_ms`, `slippage_pips`, and `max_drawdown_pct`.

**Approach:** Defined dynamic thresholds as **Mean + 2 Standard Deviations** per metric. Any trade exceeding the threshold on at least one metric is flagged as suspicious.

```python
metrics = ['lot_size', 'execution_latency_ms', 'slippage_pips', 'max_drawdown_pct']

thresholds = df[metrics].mean() + (2 * df[metrics].std())

df['is_suspicious'] = (df[metrics] > thresholds).any(axis=1).astype(int)
```

---

### Challenge 2 — Detect IP Mismatch
A geographic mismatch between a client's registered country and their IP location is a key fraud signal.

**Approach:** Normalized both country fields (strip whitespace, uppercase) before comparing to avoid false mismatches from formatting inconsistencies.

```python
df['is_ip_mismatch'] = (
    df['client_country'].str.strip().str.upper() !=
    df['ip_country'].str.strip().str.upper()
).astype(int)
```

---

### Challenge 3 — Build an Anomaly Score (0–100)
Combined multiple risk signals into a single weighted anomaly score — giving each factor a different level of importance based on its fraud relevance.

**Approach:** Normalized continuous metrics to a 0–1 scale using dynamic thresholds, then applied custom weights summing to 1.0.

| Signal | Weight | Reasoning |
|---|---|---|
| Lot Size | 30% | Unusually large trades are the strongest fraud indicator |
| IP Mismatch | 25% | Geographic anomaly is a direct fraud signal |
| Max Drawdown | 20% | Large losses may indicate manipulative trading |
| KYC Status | 15% | Unverified clients carry inherent compliance risk |
| Execution Latency | 10% | High latency can indicate system abuse |

```python
df['anomaly_score'] = (
    (df['norm_lot']      * 0.30) +
    (df['norm_latency']  * 0.10) +
    (df['norm_drawdown'] * 0.20) +
    (df['is_ip_mismatch']    * 0.25) +
    (df['is_unverified_kyc'] * 0.15)
).clip(0, 1) * 100
```

---

### Challenge 4 — Define Risk Levels
Segmented all trades into four risk tiers based on their anomaly score.

| Score Range | Risk Level |
|---|---|
| 0 – 30 | 🟢 Low |
| 30 – 60 | 🟡 Medium |
| 60 – 85 | 🟠 High |
| 85 – 100 | 🔴 Critical |

```python
bins   = [0, 30, 60, 85, 100]
labels = ['Low', 'Medium', 'High', 'Critical']

df['risk_level'] = pd.cut(df['anomaly_score'], bins=bins, labels=labels, include_lowest=True)
```

### Challenge 5 — Assign Alert Types
Labelled each trade with its **primary alert type** based on which specific signal triggered first — using priority-ordered conditions.

| Priority | Condition | Alert Type |
|---|---|---|
| 1 | Margin level < 100% | Margin Call |
| 2 | KYC Rejected or Expired | KYC Breach |
| 3 | Trade duration < 1 min | Rapid Fire Trading |
| 4 | Drawdown exceeds threshold | Large Drawdown |
| 5 | Lot size exceeds threshold | Unusual Volume |
| 6 | IP mismatch detected | IP Mismatch |
